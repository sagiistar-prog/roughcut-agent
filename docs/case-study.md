# RoughCut Agent 作品集案例

## 背景

短视频和口播内容的生产中，粗剪通常发生在正式精剪之前。创作者会积累大量手机拍摄、屏幕录制或口播片段，其中包含重复素材、空白、停顿、跑题、重说、噪音和表达不完整的内容。

传统做法需要人工反复打开视频、听内容、标时间码、筛片段，再交给剪辑软件处理。这个过程重复性强、耗时长，也很依赖剪辑者的个人经验。

## 问题

原始素材不能直接进入精剪，因为它们缺少结构化信息：

- 不知道哪些视频重复。
- 不知道每段视频讲了什么。
- 不知道哪些片段适合开头、主体、证明或结尾。
- 不知道剪切点是否会卡在人声起点。
- 不知道两个片段拼起来是否存在语义跳跃。
- 用户的剪辑偏好无法沉淀，下一次还要重新沟通。

## 目标用户

目标用户包括：

- 短视频创作者。
- 口播类内容生产者。
- 内容运营和新媒体团队。
- 需要快速把长素材整理成粗剪版本的产品、市场或教育团队。

## 核心痛点

1. 素材多，重复和低质量片段难以快速清理。
2. 口播内容需要先转成文字，才能高效筛选。
3. 粗剪不只是拼接，必须考虑节奏、气口和语义承接。
4. AI 输出如果不可复查，就难以进入真实剪辑流程。
5. 用户反馈如果只停留在对话里，下一次无法复用。

## 解决方案

RoughCut Agent 把粗剪拆成一组可执行、可复查的阶段：

1. 对原始视频做哈希去重，输出重复报告。
2. 对视频做语音识别，生成带时间码的素材索引。
3. 根据文本、质量、角色和偏好生成粗剪时间线。
4. 对剪切点应用气口规则，减少卡人声和句尾被切的问题。
5. 用 FFmpeg 根据 `timeline_review.csv` 合成粗剪视频。
6. 读取用户反馈，把“喜欢更快节奏”“保留解释更完整的片段”等偏好写入配置。

## 工作流设计

工作流的关键设计是“中间文件优先”：

- `dedupe_report.csv` 用于复核重复素材。
- `material_index.csv` 用于复核转写、时间码和片段质量。
- `timeline_review.csv` 用于复核片段顺序、剪切点和风险。
- `edit_report.md` 用于复盘最终粗剪。
- `user_preferences.yaml` 用于沉淀下一次可复用的偏好。

这种设计让 AI 不再是一次性回答，而是进入可执行的生产流程。

## 关键决策

1. 不直接修改原始素材，只读取素材并写入 `output/`。
2. 不在 GitHub 中展示真实素材，只放脱敏示例。
3. 不把不确定判断藏起来，而是写入 `risk_note`。
4. 先生成 `timeline_review.csv`，再允许 FFmpeg 合成视频。
5. 用户反馈不训练模型，只做偏好记忆和规则更新，降低实现复杂度。

## 剪辑原则

RoughCut Agent 的剪辑原则不是“尽量短”，而是“信息密度、节奏和语义连续”三者平衡：

- 开头优先选择能快速建立主题的片段。
- 主体优先选择表达完整、停顿少、声音清楚的片段。
- 删除重复表达、意思重复、空白、跑题和质量差的片段。
- 剪切点要给人声留气口，避免把话切得过紧。
- 片段之间如果逻辑跳跃，需要写入 `risk_note`，供人工复核。

## 结果

当前流程已经在真实本地素材上跑通。作品集仓库不展示真实素材，只展示脱敏样例和流程设计。

验证结果：

- 12 个去重视频。
- 43 条时间码素材行。
- 13 个粗剪片段。
- 179 秒最终粗剪版。

## 可迭代方向

1. 接入更稳定的 ASR 模型和说话人分离。
2. 支持自动生成中英双语字幕文件。
3. 增加基于 LLM 的语义排序和错别字上下文修正。
4. 支持多种剪辑风格模板，例如知识分享、产品介绍、Vlog 和带货口播。
5. 增加可视化 review 页面，让用户直接勾选保留、删除、缩短或延长片段。
6. 增加与剪辑软件的 XML/EDL 导出。

## 产品经理视角总结

RoughCut Agent 的价值不在于炫技，而在于把一段模糊的“帮我剪一下”转化为可复用、可复查、可迭代的产品流程。它保留了 AI 的理解能力，也保留了工程系统对安全、边界、配置和中间产物的控制。

这类 Agent 的真正竞争力，是把用户的高频工作流拆清楚，并让每一次使用都能让下一次更好。
